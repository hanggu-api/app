## Firebase vs Supabase-only — Análise Real

### O que o Firebase te daria a mais

**FCM (Push Notifications)**
- Já está no projeto e funcionando
- Continuar usando FCM para push é a escolha certa — Supabase não tem push nativo
- Custo: zero para volume normal

**Firebase Auth**
- Já está integrado via `loginWithFirebase()`
- Google Sign-In depende dele
- Remover seria regressão

**Crashlytics**
- Já está no projeto (`firebase_crashlytics`)
- Não tem equivalente gratuito melhor

---

### O que Firebase NÃO te daria

**Velocidade de desenvolvimento** — não, porque:
- O gargalo atual não é Firebase vs Supabase
- O gargalo é o `ApiService` god object com 3500 linhas
- Firebase não resolve acoplamento de código

**Firestore como banco** — seria regressão:
- Você já tem RLS, joins, funções SQL no Supabase
- Migrar para Firestore seria reescrever toda a lógica de dados
- Sem joins nativos, sem transações complexas

---

### Resposta direta

```
Firebase que você já usa → MANTER
├── FCM (push notifications)     ✅ sem substituto melhor
├── Crashlytics                  ✅ sem substituto gratuito melhor  
├── Google Sign-In               ✅ depende do Firebase Auth
└── Firebase Auth (token)        ✅ já integrado com Supabase

Firebase que você NÃO precisa adicionar → NÃO ADICIONAR
├── Firestore                    ❌ Supabase é superior para seu caso
├── Firebase Storage             ❌ Supabase Storage já funciona
├── Remote Config                ❌ já tem remote_config_service.dart
└── Analytics Firebase           ❌ já tem analytics_service.dart
```

---

### O que realmente acelera desenvolvimento

O ganho real vem da **modularização**, não de trocar de serviço:

- Hoje para mexer em pagamento você abre um arquivo de 3500 linhas
- Com módulos, você abre `pagamento/data/payment_repository.dart` com 200 linhas
- Qualquer dev novo entende o módulo sem precisar entender o projeto inteiro

**Conclusão:** mantém Firebase onde já está, não adiciona mais nada do Firebase, e o ganho de velocidade vem da arquitetura modular com Supabase como backend principal.

- recebe JWT no header `Authorization`
- valida com `auth.getUser()`
- retorna sempre `{ data, error }`
- nunca expõe `service_role` key
- tem no máximo 150 linhas

- tem RLS ativo
- tem `created_at` e `updated_at`
- tem índice nas colunas de busca frequente
- tem FK com `ON DELETE` definido
